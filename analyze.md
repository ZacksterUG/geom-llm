# Аналитика проекта: Веб-приложение для проверки доказательств по стереометрии

## 1. Обзор проекта
Проектируется MVP веб-приложения для пошаговой проверки геометрических доказательств (стереометрия) с интерактивной 2D-визуализацией. **Ключевая особенность**: полная делегация валидации (геометрия + логика) локальной LLM (Qwen 2.5 7B через Ollama).

## 2. Архитектурный анализ

### 2.1. Поток данных
1. **Frontend** (React/TS) генерирует состояние фигуры в формате JSON (`vertices`, `edges`, `relations`).
2. **Backend** (FastAPI) принимает шаг доказательства и состояние фигуры.
3. **LLM** (Ollama) валидирует корректность шага, опираясь на аксиомы и историю.
4. **Backend** возвращает структурированный JSON-ответ с оценкой и подсказкой.

### 2.2. Технологический стек
| Компонент | Технология | Комментарий |
| :--- | :--- | :--- |
| **Frontend** | React + Vite + TypeScript | Быстрая сборка, типобезопасность. |
| **UI Kit** | Mantine | Соблюдение ограничения (без Tailwind). |
| **Геометрия** | JSXGraph | Использование 2D аксонометрии для 3D-объектов. |
| **Backend** | FastAPI + Pydantic v2 | Асинхронность, строгая валидация схем. |
| **БД** | PostgreSQL + Liquibase | Версионирование схемы через changelog. |
| **AI** | Ollama (Qwen2.5:7b) | Локальный инференс, zero-cost API. |

## 3. Анализ требований к LLM-валидации

### 3.1. Вызовы
- **Контекст**: LLM должна "понимать" геометрические отношения (коллинеарность, перпендикулярность) из JSON-координат в 2D-проекции. Это сложная задача для модели 7B параметров.
- **Стабильность**: Требование `format: "json"` и `temperature: 0.1` критично, так как парсинг "сломанного" JSON заблокирует фронтенд.
- **Сократический метод**: Промпт должен жестко ограничивать LLM от выдачи готового решения.

### 3.2. Схема валидации (JSON)
```json
{
  "status": "ok | warning | error",
  "geometry_valid": boolean,
  "logic_valid": boolean,
  "reason": "string",
  "hint": "string (вопрос, а не ответ)",
  "referenced_theorem": "string | null",
  "next_allowed_actions": ["string"]
}
```

## 4. Анализ структуры проекта

### 4.1. Ключевые файлы
- **`docker-compose.yml`**: Оркестрация 5 сервисов. Сложность — ожидание запуска Ollama и БД.
- **`backend/app/core/ollama_client.py`**: Обертка над httpx. Реализация retry-логики и обработки таймаутов.
- **`frontend/src/components/GeometryCanvas.tsx`**: Самая сложная часть фронтенда. Необходимо реализовать:
    - Привязку кликов к координатам JSXGraph.
    - Сериализацию объектов (точки, отрезки) в плоский JSON.
    - Аксонометрическую проекцию (преобразование 3D-координат в 2D холст).

### 4.2. Упрощения для MVP
- **Сессия**: Анонимность через `localStorage` (UUID) упрощает бэкенд, убирая таблицы пользователей.
- **Кэш**: Использование `lru_cache` на бэкенде оправдано для MVP, но в проде лучше Redis.
- **Миграции**: Liquibase — тяжеловесно для MVP, но требуется по ТЗ. `001-init-tasks.sql` должен включать JSONB поле для состояния фигуры.

## 5. Риски и "подводные камни"

1. **Точность LLM**: Qwen2.5:7b может ошибаться в геометрических выводах. Рекомендуется добавить в системный промпт максимально подробные примеры (few-shot) для стереометрии 10-11 класса.
2. **Синхронизация состояния**: При изменении фигуры на фронтенде, JSON должен обновляться мгновенно. Использование `useRef` для JSXGraph и `useState` для React-состояния требует аккуратной реализации.
3. **Зависимость от Ollama**: Если модель "вылетает" или долго думает, пользователь увидит зависание. Нужен timeout на бэкенде.
4. **Аксонометрия**: Реализация "плоской" стереометрии требует математической базы (коэффициенты искажения по осям).

## 6. План реализации (Checklist)

- [ ] **Этап 1: Инфраструктура**. Создание `docker-compose.yml`, настройка `Dockerfile` для всех сервисов.
- [ ] **Этап 2: База данных**. Скрипт Liquibase, добавление тестовой задачи (например, задача о пересечении плоскостей).
- [ ] **Этап 3: Бэкенд**. Реализация эндпоинта `/api/validate`, настройка Pydantic схем, интеграция с Ollama.
- [ ] **Этап 4: Фронтенд**. Интеграция JSXGraph, создание компонентов `ProofStepInput` и `FeedbackPanel` на Mantine.
- [ ] **Этап 5: Интеграция**. Проверка полного цикла "Построение -> Шаг -> Ответ LLM".

## 7. Рекомендации по системному промпту для LLM
Чтобы LLM работала корректно, промпт должен включать:
- **Роль**: "Ты — строгий учитель геометрии, следующий учебнику Атанасяна".
- **Ограничения**: "Не давай готовое решение. Используй только предоставленные аксиомы".
- **Формат**: Явная JSON-схема внутри промпта.
- **Контекст**: Описание того, что координаты в `figure_state` являются проекцией, и как интерпретировать `relations`.

## 8. Вывод
Проект амбициозен для MVP из-за зависимости от LLM в валидации геометрии. Технический стек современный и адекватный. Главный фокус при реализации — **надежная упаковка ответа LLM** и **корректная сериализация фигуры** на фронтенде.
